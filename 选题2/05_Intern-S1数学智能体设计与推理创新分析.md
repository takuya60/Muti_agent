# XH-202627 基于 Intern-S1 的数学智能体设计与推理创新分析

## 1. 赛题理解

这个题要求基于 **Intern-S1 模型 API**，构建一个能自动求解数学题、解释推理过程、输出 JSON 结果的数学智能体。初赛偏单智能体解题，决赛扩展为多智能体协作和交互 Demo。

它不是让你训练数学大模型，而是考察你能不能围绕 Intern-S1 设计更好的数学推理流程：题型识别、解题规划、工具校验、自我修正、启发式解释和稳定 JSON 输出。

## 2. 核心难点

1. **正确率最重要**：答案正确性占 60%，答不对就很难高分。
2. **题目覆盖广**：偏微分方程、复分析、拓扑学、运筹学等 18 个子领域。
3. **不能只靠单轮调用**：赛题明确不鼓励简单提示词堆叠。
4. **JSON 必须稳定**：格式错、无法解析会按未通过处理。
5. **需要数学基础**：要能判断模型推理和答案是否合理。

## 3. 技术栈详解

| 模块 | 推荐技术 | 用途 |
|---|---|---|
| 核心模型 | Intern-S1 API | 负责数学推理和解题 |
| Agent 框架 | lagent / LangGraph / 自研状态机 | 组织题型路由、solver、verifier 流程 |
| 主开发语言 | Python | API 调用、批量处理、数学工具、评测脚本 |
| 数学工具 | SymPy / NumPy / SciPy / mpmath | 符号计算、数值验证、方程校验 |
| 优化工具 | OR-Tools / PuLP / NetworkX | 运筹学、图论、路径和优化问题辅助验证 |
| 结构化输出 | JSON Schema / Pydantic | 保证输出 JSON 合法、稳定、可解析 |
| 日志系统 | JSONL / SQLite / CSV | 保存题目、推理、校验、结果和错误信息 |
| Demo | Jupyter Notebook / Gradio / Streamlit | 展示交互解题、推理过程和统计结果 |

### 3.1 Intern-S1 API

Intern-S1 是本题指定模型。这个题主要不是训练模型，而是调用 Intern-S1，并围绕它设计更好的推理流程。

你们需要重点关注 API 调用稳定性、限流、失败重试、日志保存和批量处理。模型效果本身重要，但真正能拉开差距的是题型路由、校验和自我修正。

### 3.2 lagent / LangGraph / 状态机

官方给了 lagent 基线，建议优先研究。它适合构建 LLM-based Agent，并接入工具。

如果想要更清晰地控制流程，可以用 LangGraph 或自研状态机，把流程拆成：题型识别、规划、求解、校验、修正、解释、格式化。这样比单轮 prompt 更稳，也更符合赛题“不鼓励简单调用”的导向。

### 3.3 Python 主流程

Python 是最合适的主开发语言。它负责：

- 调用 Intern-S1 API。
- 批量读取题目。
- 保存 JSON 结果。
- 调用数学工具。
- 统计正确率和耗时。
- 生成日志和 Demo。

整个项目可以基本用 Python 完成，不一定需要复杂前端。

### 3.4 数学工具

数学工具是提升正确率的关键。

- SymPy：符号计算，适合化简、求导、积分、解方程、代入验证。
- NumPy / SciPy / mpmath：数值计算，适合近似验证和随机点代入。
- OR-Tools / PuLP：运筹学、线性规划、整数规划。
- NetworkX：图论、路径、网络流等问题。

它们不一定能解所有题，但可以帮助发现明显错误，提升稳定性。

### 3.5 JSON Schema / Pydantic

评分要求结构化 JSON 输出，格式错误会直接影响判分。Pydantic 可以在本地校验模型输出是否合法。

建议模型输出后不要直接提交，而是先过一层格式检查。如果字段缺失、类型错误或 answer 为空，就触发格式修复或重新生成。

### 3.6 日志系统

每道题都应该保存完整日志，包括题目、题型、使用的 solver、模型回答、校验结果、最终 JSON、重试次数和错误信息。

日志的作用：

- 方便复现。
- 方便排查错误。
- 方便展示系统设计。
- 避免批量处理时中断后全部丢失。

### 3.7 Demo 展示

决赛可以用 Gradio、Streamlit 或 Notebook。建议展示：

- 输入题目。
- 题型识别结果。
- 解题计划。
- 调用的 solver / verifier。
- 校验过程。
- 最终答案。
- 学习启发式解释。
- JSON 输出。

如果时间有限，Streamlit 或 Gradio 比完整 Web 前端更合适。

## 4. 推荐技术路线

### 阶段一：跑通 Intern-S1 API

1. 获取 API token。
2. 写 Python 脚本调用 Intern-S1。
3. 输入一道简单数学题。
4. 输出答案和简短推理。
5. 保存请求和响应日志。

目标：先能稳定调用模型。

### 阶段二：定义 JSON 输出格式

建议统一输出：

```json
{
  "problem_id": "001",
  "domain": "complex_analysis",
  "answer": "...",
  "reasoning_summary": "...",
  "confidence": 0.82,
  "verification_status": "passed"
}
```

用 Pydantic 校验字段是否齐全、类型是否正确。格式不合法就自动修复或重试。

### 阶段三：批量处理题目

1. 读取组委会题目数据。
2. 循环调用智能体。
3. 每题单独保存结果和日志。
4. API 失败自动重试。
5. 支持断点续跑。
6. 汇总正确率、耗时、失败原因。

这一步非常重要，因为初赛要处理全部题目。

### 阶段四：题型识别与路由

不同数学题走不同策略：

| 题型 | 策略 |
|---|---|
| 代数 / 方程 | Intern-S1 推理 + SymPy 代入验证 |
| 微积分 | 求导、积分、极限工具校验 |
| 偏微分方程 | 方程结构分析 + 代入检查 |
| 复分析 | 定理识别 + 条件检查 |
| 运筹学 | 建模 + OR-Tools / PuLP 验证 |
| 图论 | NetworkX 辅助验证 |
| 证明题 | Verifier Agent 检查逻辑链 |

题型路由是这个题最自然的创新点。

### 阶段五：过程校验和自我修正

流程建议：

```text
Solver 输出答案
  ↓
工具或 Verifier 校验
  ↓
如果失败，提取失败原因
  ↓
反馈给 Solver 重新推理
  ↓
最多重试 N 次
  ↓
输出最可信答案
```

校验方式：

- 符号校验：化简、求导、代入原方程。
- 数值校验：随机取点代入，检查误差。
- 逻辑校验：另一个 Agent 审核推理是否跳步或误用定理。

### 阶段六：生成学习启发式解释

解释不要只是堆公式，而要像讲题：

1. 题目关键条件是什么。
2. 为什么选这个方法。
3. 用到了哪个定理或技巧。
4. 哪一步最容易错。
5. 最终答案是什么。

这部分能提升主观评分和展示效果。

### 阶段七：决赛多智能体扩展

推荐 Agent 分工：

| Agent | 职责 |
|---|---|
| Router Agent | 判断题型 |
| Planner Agent | 制定解题计划 |
| Solver Agent | 调用 Intern-S1 解题 |
| Tool Agent | 调用 SymPy / SciPy / OR-Tools |
| Verifier Agent | 检查答案和推理 |
| Explainer Agent | 生成启发式讲解 |
| Formatter Agent | 保证 JSON 格式 |

决赛 Demo 展示一题从输入到路由、求解、校验、修正、解释、JSON 输出的全过程。

## 5. 可做创新点

1. 题型路由：不同数学领域调用不同 solver。
2. 过程校验：符号计算、数值验证、逻辑审核结合。
3. 自我修正：校验失败后反馈错误原因并重推。
4. 稳定 JSON 输出：格式校验和自动修复。
5. 学习启发式解释：把推理过程转成教学讲解。
6. 多智能体协作：Router、Solver、Verifier、Explainer、Formatter 分工。

## 6. 可行性判断

| 维度 | 判断 |
|---|---|
| 技术难度 | 中等 |
| 算力需求 | 低 |
| 展示效果 | 中等偏强 |
| AI / Agent 味道 | 强 |
| 获奖潜力 | 较高 |
| 主要风险 | 数学题难、正确率压力、API 稳定性、JSON 格式错误 |

## 7. 适合的团队

适合：

- 有 Python 工程能力。
- 有数学基础较强的成员。
- 想做 AI Agent，但不想训练大模型。
- 愿意打磨正确率、日志、评测和校验流程。

不适合：

- 数学基础较弱。
- 只想做界面，不想打磨推理正确率。
- 不愿意处理 JSON、日志、重试、评测等细节。

## 8. 结论

这是新增后非常值得考虑的题。它比太空智算风险低很多，也比荣耀题边界更清楚，同时很贴合 AI Agent。最大问题是正确率和数学能力，如果团队有数学基础，它可以作为强备选，甚至可以和车路云竞争第一选择。
