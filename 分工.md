# 领域知识个性化生成与多智能体协同决策系统：项目结构与团队分工方案

## 一、项目定位

建议将本赛题落地为：

> 面向人工智能 / 机器学习实训场景的个性化学习资源生成与多智能体协同决策系统。

也可以替换为 Python 数据分析、前端开发、Linux 运维、工业互联网等垂直领域，但建议优先选择团队最熟悉、资料最容易整理、评测最容易设计的方向。当前最推荐“机器学习入门实训”或“Python 数据分析实训”，因为知识点清晰、资料充足、容易设计学习者画像和测试题。

系统核心闭环：

```text
学习者画像输入
  ↓
学情诊断 Agent：识别基础、目标和知识盲区
  ↓
知识检索 Agent：从专业知识库中检索依据和实操规范
  ↓
资源生成 Agent：生成讲义、实操指南、分阶测试题
  ↓
审核纠偏 Agent：检查幻觉、专业错误和难度匹配
  ↓
反馈决策 Agent：根据答题结果和学习反馈调整路径
  ↓
可视化报告与个性化资源输出
```

这个项目不要做成普通聊天机器人，而要突出三点：

1. **有知识库支撑**：生成内容必须能追溯到领域资料，降低幻觉。
2. **有多 Agent 协同**：至少 3 个 Agent，最好实现 5 个 Agent 的完整闭环。
3. **有个性化差异**：同一学习目标下，不同基础的学习者应得到不同难度、不同路径、不同练习。

## 三、核心模块说明

### 1. 学习者画像模块

负责接收并维护学习者信息，例如：

```json
{
  "learner_id": "student_001",
  "background": "大二学生，会 Python 基础，但没系统学过机器学习",
  "goal": "掌握逻辑回归分类模型实操",
  "test_scores": {
    "python": 80,
    "linear_algebra": 45,
    "ml_basic": 50,
    "model_evaluation": 40
  },
  "preferred_style": "案例驱动",
  "available_time": "每周 4 小时"
}
```

至少准备 3 类画像：

| 类型 | 特征 | 系统应生成的资源特点 |
|---|---|---|
| 基础薄弱型 | 会一点编程，数学和机器学习基础弱 | 更多类比解释、基础补课、分步实操 |
| 普通基础型 | 有 Python 和部分数学基础，但缺少项目经验 | 标准讲义、完整实验流程、分阶练习 |
| 进阶提升型 | 有基础，希望做项目或挑战任务 | 简洁理论回顾、项目任务、扩展阅读和进阶题 |

### 2. 多 Agent 工作流模块

建议实现 5 个 Agent：

| Agent | 核心职责 | 输入 | 输出 |
|---|---|---|---|
| 学情诊断 Agent | 分析学习者基础、目标、薄弱知识点 | 学习者画像、测试结果 | 知识盲区、推荐难度、学习目标拆解 |
| 知识检索 Agent | 从知识库中检索相关概念、实验步骤和规范 | 目标知识点、学习者层级 | 证据片段、引用来源、关键知识点 |
| 资源生成 Agent | 生成定制讲义、实操指南、分阶测试题 | 学情诊断结果、检索证据 | 个性化学习资源初稿 |
| 审核纠偏 Agent | 检查专业错误、幻觉、难度不匹配、覆盖不足 | 资源初稿、知识库证据、评分规则 | 审核意见、修改建议、是否通过 |
| 反馈决策 Agent | 根据答题结果或学习反馈决定下一步 | 学习反馈、答题正确率、历史状态 | 降维解释、补充练习或进阶挑战 |

注意：这里可以保留“知识检索 Agent / 领域专家 Agent”的系统角色，但不建议在团队人力中单独设置“领域知识工程师”。领域知识整理、知识库构建、评测样例设计应拆分到 RAG、评测、PM 和 Agent 负责人中共同完成。

### 3. RAG 知识库模块

RAG 模块要解决“生成内容有依据”的问题，建议做到：

1. 文档清洗：把教材片段、实验步骤、官方文档整理成 Markdown / JSON。
2. 文档切片：按知识点、实验步骤、概念解释进行分块。
3. 向量检索：用 FAISS / Chroma 做本地向量库。
4. 证据约束：资源生成 Agent 必须基于检索证据生成。
5. 来源追溯：最终资源展示“依据来自哪些知识库片段”。

### 4. 个性化资源生成模块

至少生成 3 类资源，满足赛题要求：

| 资源类型 | 内容要求 | 示例 |
|---|---|---|
| 定制讲义 | 根据学习者基础解释核心概念 | 逻辑回归原理、损失函数、适用场景 |
| 实操指南 | 给出可执行的步骤 | 数据加载、训练模型、评估准确率、可视化结果 |
| 分阶测试题 | 按难度出题并给解析 | 基础选择题、代码填空题、项目思考题 |

增强项：学习路径图、错题解析、进阶挑战任务、学习反馈后的二次生成。

### 5. 可视化展示模块

演示界面建议包含：

1. 左侧：学习者画像输入、测试结果输入、学习目标选择。
2. 中间：多 Agent 协同流程可视化，显示每个 Agent 的状态和输出摘要。
3. 右侧：生成的讲义、实操指南、测试题。
4. 下方：知识盲区定位、难度匹配曲线、知识点覆盖率、学习路径规划图。
5. 反馈区：学习者选择“太难 / 太简单 / 想看案例 / 已完成测试”，系统触发二次决策。

### 6. 评测模块

比赛评分明确强调实用价值和量化指标，建议至少实现：

| 指标 | 目标值 | 评测方式 |
|---|---:|---|
| 专业知识谬误率 / 幻觉率 | < 5% | 人工标注 + LLM-as-Judge + 知识库证据核对 |
| 学习者画像-资源难度适配准确率 | ≥ 85% | 规则评分 + 多画像对比测试 |
| 核心知识点覆盖率 | ≥ 90% | 知识点清单匹配 + 审核 Agent 打分 |
| Agent 流程成功率 | ≥ 95% | 自动化测试统计 |
| 资源结构完整率 | ≥ 95% | JSON Schema / Pydantic 校验 |

## 四、团队分工建议：6-7 人版本

团队不建议设置单独的“领域知识工程师”。原因是：

1. 这个岗位容易变成单纯整理资料，和系统工程脱节。
2. 领域知识本身应该融入 RAG、评测、Prompt、演示材料，而不是由一个人单独负责。
3. 比赛更看重系统闭环、技术创新、可视化和评测指标，领域资料整理可以拆分给多个技术角色共同完成。

推荐 6-7 人按如下方式分工。

### 角色总览

| 编号 | 角色 | 人数 | 主要目标 |
|---|---:|---:|---|
| 1 | 项目负责人 / 产品与方案统筹 | 1 | 把控赛题要求、进度、方案文档、最终提交材料 |
| 2 | Agent 工作流工程师 | 1 | 设计多 Agent 架构、状态流转、Prompt 和协同闭环 |
| 3 | RAG 与知识库工程师 | 1 | 构建知识库、检索链路、证据引用和知识溯源 |
| 4 | 后端与数据接口工程师 | 1 | 搭建 API、数据模型、业务服务和系统集成 |
| 5 | 前端与可视化工程师 | 1 | 完成 Demo 界面、Agent 流程可视化和报告展示 |
| 6 | 评测与测试工程师 | 1 | 设计测试集、评测指标、自动化测试和质量报告 |
| 7 | 演示材料与交互体验负责人 | 0-1 | 负责演示脚本、PPT、视频、资源排版和用户体验打磨 |

如果团队是 **6 人**：第 7 个角色由项目负责人、前端工程师、评测工程师共同承担。  
如果团队是 **7 人**：第 7 个角色独立出来，专门提升展示效果和材料质量。

## 五、各岗位详细分工与技术栈

### 1. 项目负责人 / 产品与方案统筹

**核心职责**

- 明确项目选定的垂直领域和目标用户。
- 拆解赛题评分标准，确保作品覆盖完整性、创新性、用户体验、实用价值四类要求。
- 维护需求清单、里程碑、每周任务和风险列表。
- 统筹作品设计方案、部署说明、测试报告、PPT、演示视频等提交材料。
- 组织每周集成，避免各模块各做各的。

**主要交付物**

- 项目需求说明。
- 作品设计实现方案。
- 比赛提交清单。
- 项目进度表。
- 演示视频整体脚本初稿。

**建议技术栈 / 能力要求**

- Markdown / Office / PPT。
- 基本产品原型能力：墨刀、Figma、ProcessOn、draw.io 均可。
- 基本 Git 协作能力。
- 能看懂 API、Agent、RAG、评测指标，不一定负责具体实现。

**重点关注**

- 不要让项目偏成“普通 AI 课程生成器”。
- 每个功能都要能对应赛题评分点。
- 确保最后能提交源码、部署说明、测试数据、演示视频和设计文档。

### 2. Agent 工作流工程师

**核心职责**

- 设计多智能体协同架构。
- 实现“分析-生成-校验-决策”的闭环流程。
- 编写诊断、检索、生成、审核、反馈等 Agent 的 Prompt。
- 实现审核失败后的重生成机制。
- 设计 Agent 间共享状态和结构化输出格式。

**主要交付物**

- `agents/workflow.py`
- `agents/state.py`
- 各 Agent 实现文件。
- Prompt 模板。
- Agent 流程日志，供前端可视化展示。

**建议技术栈**

- Python。
- LangGraph，优先推荐；也可用 AutoGen、CrewAI 或自研状态机。
- Pydantic / JSON Schema。
- OpenAI API / Qwen API / DeepSeek API / GLM API 等。
- Prompt Engineering。

**重点关注**

- Agent 之间不能只是顺序调用大模型，要体现明确职责。
- 审核 Agent 必须能提出具体修改意见。
- 工作流要保留中间状态，方便前端展示“每个 Agent 做了什么”。

### 3. RAG 与知识库工程师

**核心职责**

- 整理垂直领域知识库，不单独做人肉“领域专家”，而是把资料工程化。
- 完成文档清洗、切片、向量化、检索和证据组装。
- 为生成 Agent 提供可靠的检索上下文。
- 为审核 Agent 提供专业依据，帮助判断幻觉和错误。
- 建立知识点依赖关系，例如“Python 基础 → NumPy → 线性代数 → 逻辑回归 → 模型评估”。

**主要交付物**

- `data/knowledge_base/`
- `data/knowledge_graph.json`
- `rag_engine/document_loader.py`
- `rag_engine/vector_store.py`
- `rag_engine/retriever.py`
- 知识库来源说明。

**建议技术栈**

- Python。
- LangChain / LlamaIndex。
- FAISS / Chroma。
- sentence-transformers / bge / text-embedding API。
- Markdown / JSON 数据清洗。
- NetworkX，可用于知识图谱和学习路径展示。

**重点关注**

- 知识库宁可小而准，不要大而乱。
- 每个知识点最好有对应概念解释、实操步骤和测试题依据。
- 最终资源最好能展示引用来源，突出“高保真”和“可追溯”。

### 4. 后端与数据接口工程师

**核心职责**

- 搭建后端 API 服务。
- 对接前端、Agent 工作流、RAG、评测模块。
- 管理学习者画像、生成记录、反馈记录、评测结果。
- 统一数据结构，保证各模块输入输出一致。
- 提供一键运行或本地部署能力。

**主要交付物**

- `backend/main.py`
- `backend/api/`
- `backend/services/`
- `schemas/`
- `deploy/`
- `.env.example`

**建议技术栈**

- Python。
- FastAPI，优先推荐；Flask 也可。
- Pydantic。
- SQLite / PostgreSQL；比赛 Demo 初期用 SQLite 或 JSON 即可。
- Uvicorn。
- Docker / docker-compose，后期用于打包部署。

**重点关注**

- 接口要稳定，不能前端每次点生成都随机报错。
- API 返回结构要适合前端展示，不能只返回一整段文本。
- 注意不要把真实 API Key 写进代码或提交仓库。

### 5. 前端与可视化工程师

**核心职责**

- 实现比赛演示界面。
- 展示学习者画像输入、多 Agent 流程、生成资源和评测报告。
- 可视化知识盲区、难度匹配曲线、学习路径图。
- 设计反馈交互，让系统能进行二次生成或路径调整。

**主要交付物**

- `frontend/app.py`
- `frontend/components/`
- 学情报告页面。
- Agent 流程可视化页面。
- 个性化资源展示页面。
- 反馈迭代页面。

**建议技术栈**

- 快速 Demo 路线：Streamlit / Gradio。
- 更完整产品路线：Vue 3 + Vite + ECharts + Axios。
- 图表：ECharts / Plotly / Altair。
- 流程图：Mermaid / Graphviz / NetworkX 可视化。

**重点关注**

- 比赛展示时，评委需要看到“过程”，不是只看到最终文本。
- 同一目标下展示 2-3 个不同学习者的输出对比，能直接体现个性化。
- 生成资源的排版要清晰，讲义、实操、测试题要分区展示。

### 6. 评测与测试工程师

**核心职责**

- 设计测试数据和测试用例。
- 实现幻觉率、难度适配率、知识点覆盖率等评测指标。
- 编写自动化测试，验证多 Agent 流程、RAG 检索、结构化输出是否稳定。
- 输出测试报告，支撑比赛材料中的“实用价值”。

**主要交付物**

- `evaluation/metrics.py`
- `evaluation/llm_judge.py`
- `evaluation/test_cases.json`
- `evaluation/report_generator.py`
- `tests/`
- `docs/test_report.md`

**建议技术栈**

- Python。
- pytest。
- Pandas。
- Pydantic 数据校验。
- LLM-as-Judge。
- 简单规则评分与人工标注表格。

**重点关注**

- 至少准备 3 组差异化学习者画像，建议扩展到 10-30 条测试样例。
- 指标要能解释，不要只有模型打分。
- 测试报告中要展示不同学习者生成结果的差异，以及审核纠偏前后的质量提升。

## 七、建议技术路线

### 快速可落地版本

适合比赛初期快速做出可演示系统：

```text
Python + FastAPI + LangGraph + Chroma/FAISS + Streamlit + SQLite/JSON + pytest
```

优点：开发快、依赖少、适合学生团队快速集成。  
缺点：前端视觉效果不如完整 Web 项目。

### 展示增强版本

适合后期打磨：

```text
Python + FastAPI + LangGraph + Chroma/FAISS + Vue 3 + ECharts + Docker
```

优点：界面更像完整产品，用户体验更好。  
缺点：前后端联调成本更高。

建议先做快速可落地版本，确保闭环跑通，再决定是否升级前端。

## 八、推荐开发里程碑

### 第一阶段：项目定型与最小闭环

时间建议：第 1-2 周。

目标：跑通最小 Demo。

- 确定垂直领域。
- 准备 3 组学习者画像。
- 准备一小批知识库资料。
- 实现 3 个基础 Agent：诊断、生成、审核。
- 前端能输入画像并展示生成结果。

### 第二阶段：多 Agent 完整闭环

时间建议：第 3-5 周。

目标：实现赛题要求的完整流程。

- 增加知识检索 Agent 和反馈决策 Agent。
- 打通 RAG 检索。
- 实现审核失败后的重生成。
- 输出讲义、实操指南、分阶测试题三类资源。
- 保存 Agent 中间过程。

### 第三阶段：可视化与评测体系

时间建议：第 6-8 周。

目标：让系统有可展示、可解释、可评分的能力。

- 增加 Agent 流程可视化。
- 增加知识盲区、难度匹配曲线、学习路径图。
- 完成幻觉率、难度适配率、知识点覆盖率评测。
- 准备更多测试样例。

### 第四阶段：材料打磨与比赛提交

时间建议：提交前 2-3 周。

目标：形成完整参赛作品包。

- 完成设计实现方案。
- 完成部署说明。
- 完成测试报告。
- 录制 10 分钟以内演示视频。
- 检查源码、测试数据、文档、PPT 是否齐全。

## 九、比赛评分点对应策略

| 评分项 | 分值 | 项目应对策略 |
|---|---:|---|
| 作品完整性 | 30 | 做完“画像 → Agent 调度 → RAG → 生成 → 审核 → 反馈 → 报告”的闭环 |
| 技术创新性 | 25 | 强调多 Agent 交叉验证、审核纠偏、知识溯源、动态反馈 |
| 用户体验 | 15 | 做可视化界面，展示流程、报告和资源，不让系统黑盒运行 |
| 实用价值 | 30 | 提供知识库、≥3 类学习者画像、评测指标和测试报告 |

## 十、最小可行版本范围

如果时间紧，优先完成以下内容：

1. 一个明确领域：例如机器学习入门实训。
2. 一个小型知识库：10-30 个知识点文档切片。
3. 三类学习者画像：基础薄弱、普通基础、进阶提升。
4. 至少 4 个 Agent：学情诊断、知识检索、资源生成、审核纠偏。
5. 三类资源：定制讲义、实操指南、分阶测试题。
6. 一个反馈机制：根据答题正确率生成“降维解释”或“进阶挑战”。
7. 一个演示界面：能展示 Agent 流程和个性化结果。
8. 一份测试报告：说明幻觉率、适配率、覆盖率如何计算。

## 十一、最终建议

这个项目最重要的不是模型规模，而是系统设计。团队应该优先把以下闭环做扎实：

```text
差异化学习者画像
  + 专业知识库约束
  + 多 Agent 协同审核
  + 个性化资源生成
  + 反馈驱动的动态调整
  + 可视化评测报告
```

分工上不建议设置单独“领域知识工程师”。更合理的方式是：

- 项目负责人把控场景和比赛要求。
- RAG 工程师把领域资料变成可检索知识库。
- Agent 工程师让模型按证据生成和被审核。
- 评测工程师验证内容是否准确、适配、覆盖完整。
- 前端和演示负责人把过程与结果清晰展示出来。

这样更符合比赛评分逻辑，也更容易做出一个能演示、能解释、能量化评测的完整作品。
